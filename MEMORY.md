- 最新状态（截至2026-02-24）：利差已转正，倒挂信号显著弱化

## 部署和网站管理经验教训 (2026-02-26)

### 问题总结
**GitHub Pages 部署失败的根本原因**：
- 本地工作分支为 `master`，但 GitHub 仓库的默认分支和 GitHub Pages 部署源为 `main`
- 导致推送的更改无法在 GitHub Pages 网站上显示
- 用户在 GitHub 网页上看不到更新的文件

### 解决方案
1. **分支识别**：始终确认 GitHub 仓库的默认分支名称（通过 `git remote show origin` 或 GitHub API）
2. **正确推送**：确保所有更改推送到正确的分支（通常是 `main` 而不是 `master`）
3. **目录结构清理**：保持简洁的标准 GitHub Pages 结构：
   ```
   /
   ├── index.html
   ├── style.css  
   └── reports/
       ├── YYYY-MM-DD.html
   ```
4. **部署验证**：推送后立即检查 GitHub 网页和实际网站是否同步更新

### 预防措施
- 在开始任何网站更新前，先确认目标分支
- 使用 `git branch -r` 查看远程分支
- 推送后验证 GitHub 网页界面是否显示更新
- 如果网站不更新，立即检查分支配置而非重复推送

### 关键命令参考
```bash
# 检查远程分支
git remote show origin

# 切换到正确的分支
git checkout main

# 推送到正确分支
git push origin main
```

此经验教训将帮助避免未来出现类似的部署问题。
